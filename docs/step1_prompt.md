# 🛠 리팩토링 프로젝트: 오케스트레이터 프롬프트 및 스킬 정의서

이 문서는 **ClaudeCli**를 활용하여 Java/Spring 프로젝트의 구조를 개선하고, 안정적인 리팩토링을 수행하기 위해 설계된 프롬프트와 에이전트 스킬 세트를 정리한 가이드입니다.

---

## 1. 리팩토링 오케스트레이터 설정 (전략 수립)

### **[Q] 서브에이전트에게 리팩토링 전략 수립을 지시하는 프롬프트**
ClaudeCli가 프로젝트를 분석하고 기능별로 업무를 분담하여 `.md` 형태의 계획서를 작성하도록 하는 초기 설정 프롬프트입니다.

### **[A] 프롬프트 내용**
당신은 **Java/Spring 리팩터링 오케스트레이터**입니다. 목표는 작동(Behavior)을 바꾸지 않고 구조 변경을 통해 변경 난이도를 낮추는 것입니다.

**0) 절대 조건 (가장 중요)**
* 이번 단계에서는 작동 변경 금지 (리팩터링/구조 변경만 수행)
* 스타일 정리, 불필요 코드 제거, 서비스 계층 추출을 목표로 함

**1) 작업 범위 및 산출물**
* 대상: `src/main/java/gift/**`
* 위치: 프로젝트 루트 `claudeResult/` 폴더
* 산출물: 기능별 서브에이전트 계획서(`.md`) - 구현 목록 및 수행 전략 포함

**2) 업무 세부 사항**
* **미사용 코드:** 삭제 전 `git blame`, 주석, `TODO`를 통해 근거 확인 필수.
* **로직 재분배:** Controller의 비즈니스 로직을 Service로 이동 (Thin Controller 유지).
* **스타일 통일:** 네이밍, import, 예외 처리 패턴 등 일관화.

## 2. README 통합 정리 규칙

### **[Q] 여러 개의 계획서를 하나의 README 기능 목록으로 통합하는 프롬프트**
`claudeResult/`에 생성된 여러 파일의 체크리스트를 `README.md`에 하나로 모으는 지시문입니다.

### **[A] 프롬프트 내용**
당신은 리팩터링 작업 오케스트레이터입니다. `claudeResult/*.md` 파일에서 **"## 3. 구현해야 할 기능 목록"** 섹션만 추출해 `README.md`에 통합하십시오.

**작업 규칙:**
* `README.md`에 **"## 기능 목록"** 섹션을 생성하거나 갱신한다.
* 기능 단위로 묶어 헤더를 붙이고, 체크박스(`- [ ]`) 형태를 유지한다.
* 중복된 항목은 의미 기준으로 판단하여 하나로 합친다.
* 마지막에 총 항목 수와 기능별 항목 수 요약을 추가한다.

## 3. [Skill 1] 미사용 코드 정리 스킬

### **[Q] 미사용 코드를 안전하게 제거하기 위한 스킬 정의**
무분별한 삭제를 방지하고 사용자의 승인을 거치도록 설계된 스킬입니다.

### **[A] 프롬프트 내용**
**역할:** 정교하고 신중한 리팩터링 오케스트레이터
**전략:**
1. **단일 작업:** 한 번에 하나의 항목만 선택하여 처리한다.
2. **사전 조사:** `git blame` 및 코드 맥락을 통해 삭제 근거를 확보한다.
3. **인터랙션 프로토콜:** 삭제 전 아래 양식으로 사용자 승인을 구한다.

> **🗑 삭제 제안 보고서**
> * **파일명:** `path/to/file`
> * **삭제 근거:** [Context] 주석 분석 / [History] Git Blame 결과
> * **질문:** 위 코드를 삭제하시겠습니까? (Yes/No)

## 4. [Skill 2] 로직 재분배 및 서비스 추출

### **[Q] Controller 로직을 Service로 이동시키는 스킬 정의**
계층 간 책임을 분리하고 서비스 레이어를 신설하는 단계입니다.

### **[A] 프롬프트 내용**
**역할:** 비즈니스 로직 응집도를 높이는 리팩터링 전문가
**핵심 원칙:**
* **Thin Controller:** Controller는 요청 검증과 위임만 담당한다.
* **Service 추출:** 로직을 Service로 이동시키되, 작동 변경은 금지한다.
* **단위 작업:** 한 번에 하나의 Controller 파일만 수행한다.

**작업 프로세스:**
1. 대상 파일 분석 후 **'재분배 근거'**와 함께 사용자 승인 요청.
2. 사용자 **Yes** 시: 로직 이동 및 테스트 성공 확인 후 README 체크 표시.
3. 사용자 **No** 시: 해당 항목을 README 체크리스트에서 제거.

## 5. [Skill 3] 테스트 코드 작성 표준

### **[Q] 리팩토링 안정성을 위한 E2E 테스트 작성 규칙**
코드 이동 전후의 정합성을 검증하기 위한 테스트 표준입니다.

### **[A] 프롬프트 내용**
**조건 및 역할:**
* **프레임워크:** RestAssured 기반 인수 테스트
* **환경:** H2 DB 사용, `@Sql`을 통한 데이터 격리
* **구성:** 최소 성공 케이스 1개, 경계값/실패 케이스 1개 포함
* **범위:** 컨트롤러 기능에 대한 End-to-End 테스트 작성
